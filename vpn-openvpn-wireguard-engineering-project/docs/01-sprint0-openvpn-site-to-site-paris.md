<h1> 🏁 Sprint 0: Deployment of the OpenVPN Central Hub (Paris) & Multi-Site Interconnection  </h1>

## Objectives of the sprint

The aim of this first sprint is to set up the basic infrastructure for the virtual private network by setting a VPN tunnel between Tokyo/NY clients and the primary site (Paris Montrouge).

- Deploy OpenVPN on  the primary server site (Paris Montrouge) 
- Connect Tokyo & New York clients to the central site  via an unsecured network (Internet)
- Check routing and inter-site communication

## Technical Specifications
* **Protocol**: OpenVPN (VPN SSL/TLS) over UDP transport layer.
* **Security**: Strong authentication via public key infrastructure (X.509 PKI) and asymmetric encryption for key exchange (2048-bit Diffie-Hellman).
* **Addressing Architecture**:
  * **VPN Network (tun0)**: `10.9.1.0/24`
  * **IP WAN Publique (Paris and Auber)** : 88.162.141.79
  * **Primary Site (Paris)**: VPN IP `10.9.1.1` | Physical LAN: `192.168.1.197` | Inter-site link: `192.168.100.200`
  * **Tokyo Site**: Fixed VPN IP `10.9.1.2` | Remote Local LAN: `172.20.10.0/28` (VM: `172.20.10.3`)
  * **New York Site**: Fixed VPN IP `10.9.1.3` | Remote Local LAN: `172.20.10.0/28` (VM: `172.20.10.4`)
  * **Backup Site (Aubervilliers)**: No active tunnel at this stage | Inter-site: `192.168.100.210`


## 🔐 2. PKI Initialisation 
The ideal authentication solution to use for implementing an OpenVPN tunnel is using X.509 certificates. This significantly improves security by ensuring that each peer/entity (here the servers and the clients) proves its identity using a chain of trust.

Certificates will be generated by the PKI tools openssl.

The full PKI setup (CA creation, key generation, certificate signing, installation steps) is documented here:

[Authentication via PKI certificate](pki-certificate-authentication.md)

## 3. Topologie Réseau, NAT et Sécurité Périmétrique
Pour permettre aux agences distantes d'initier la connexion vers le serveur central situé derrière une Box Internet grand public, une règle de redirection de port (DNAT) et un durcissement du pare-feu local ont été configurés.
Redirection de Port (NAT Box Paris)
    IP WAN Publique : 88.162.141.79
    Règle appliquée : Port Externe UDP/32768 ➔ Port Interne Destination 192.168.1.197:1194

Ouverture du Pare-feu (UFW sur Paris Montrouge)
Par défaut, le port OpenVPN standard était bloqué. L'autorisation explicite du flux UDP a été validée :
```bash
ufw allow 1194/udp
```

<h2> Démarrage des Services OpenVPN  </h2>
Les fichiers de configuration épurés (sans commentaires de développement) sont stockés dans le dossier racine configs/openvpn/. 
Le chargement est assuré par :

# Sur le serveur Paris
```bash
systemctl start openvpn@server-parismont
```
# Sur les clients distants
```bash
systemctl start openvpn@client-tokyo-parismont
systemctl start openvpn@client-NY-parismont
```

## 4. Gestion de l'Adressage Statique et Routage Avancé
Fixation des IP VPN via le mécanisme CCD (Client-Config-Dir)

Pour garantir la persistance des règles de routage, chaque client reçoit une IP fixe basée sur le Common Name (CN) de son certificat X.509.
Dans la configuration du serveur Paris, la directive suivante a été insérée : client-config-dir /etc/openvpn/ccd.

Fichiers de configuration CCD créés sur Paris :
    Fichier /etc/openvpn/ccd/client-tokyo : ifconfig-push 10.9.1.2 255.255.255.0
    Fichier /etc/openvpn/ccd/client-NY : ifconfig-push 10.9.1.3 255.255.255.0
